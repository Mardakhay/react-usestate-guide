01 Concept

What is State?

Before we touch any code, let's understand the concept. State is the memory of a component. It's data that the component remembers, and when that data changes, the component updates what it shows on screen.

🧠

Memory

State is what a component "remembers" between renders

🔄

Reactivity

When state changes, the component automatically re-renders

📦

Private

Each component owns its own state — isolated and independent

🎯

vs Props

Props come from outside. State lives inside the component

💡 Real-world analogy

Think of your TV's volume. Every time you press +, the number on screen changes. That number is "state" — it's stored, it changes, and the screen updates automatically when it does.

⚠️ The Problem Without State

Why can't we just use a regular variable? Let's see what happens…

❌ broken — won't work

Counter.jsx
function Counter() {
  let count = 0; // 😞 plain variable

  function handleClick() {
    count = count + 1;
    // ↑ count changes in memory
    // but React has NO idea!
    // Screen never updates 😢
  }

  return (
    <div>
      {count} // stuck at 0 forever
      <button onClick={handleClick}>
        +1
      </button>
    </div>
  );
}

✅ correct — works perfectly

Counter.jsx
import { useState } from 'react';

function Counter() {
  // ✨ useState tells React to watch this!
  const [count, setCount] = useState(0);

  return (
    <div>
      {count} // updates every time!
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
    </div>
  );
}
💡

React doesn't know about regular JS variables. You must use useState so React can "watch" the value and update the screen when it changes.

02 Core Hook

The useState Hook

A hook is a special React function that lets you "hook into" React features from inside your components. useState is the most important one.

🔬 Anatomy of useState

syntax.js
// Pattern:
const [value, setValue] = useState(initialValue);
//     ↑         ↑                    ↑
//  current   function to           starting
//   value    update value           value


// Real examples:
const [count,   setCount]   = useState(0);
const [name,    setName]    = useState('');
const [isOpen,  setIsOpen]  = useState(false);
const [items,   setItems]   = useState([]);
const [user,    setUser]    = useState(null);
// Convention: setter name = "set" + variable name
🎯

The [ ] syntax is called array destructuring. useState returns an array of exactly 2 things — the current value and the update function. You're just naming them when you destructure.

03 Data Types

State with Different Data Types

useState works with any JavaScript data type. Here's how to use each one correctly.

🔢 Number & String

types.jsx
// Numbers
const [count,    setCount]   = useState(0);
const [price,    setPrice]   = useState(9.99);

// Strings
const [name,     setName]    = useState('');
const [title,    setTitle]   = useState('Hello');

// Updating them:
setCount(5);            // set to a specific value
setCount(count + 1);    // add 1 to current value
setName('Ali');         // change the string

✅ Boolean — for on/off toggles

toggle.jsx
const [isOpen, setIsOpen] = useState(false);
const [isDark, setIsDark] = useState(true);

// Toggle: flip true→false or false→true
setIsOpen(!isOpen);          // simple way
setIsDark(prev => !prev);   // ✅ safer way (more on this later)

// Using boolean state in JSX:
return (
  <div>
    {isOpen && <Modal />}   // show if true
    {isDark ? 'Dark' : 'Light'}  // conditional text
  </div>
);
isOpen: false

📋 Array State — the rules

Arrays need special attention. Never mutate (modify) the array directly! Always create a new array.

arrays.jsx
const [todos, setTodos] = useState([]);

// ❌ NEVER do this — React won't detect the change:
todos.push(newItem); // mutates original — React ignores it!

// ✅ ADD an item — spread operator creates a NEW array:
setTodos([...todos, newItem]);

// ✅ REMOVE an item — filter creates a NEW array:
setTodos(todos.filter(t => t.id !== idToRemove));

// ✅ UPDATE an item — map creates a NEW array:
setTodos(todos.map(t =>
  t.id === targetId
    ? { ...t, completed: true }  // change this one
    : t                              // keep others same
));

🗂️ Object State — spread to update

objects.jsx
const [user, setUser] = useState({
  name: '',
  email: '',
  age: 0
});

// ✅ Update ONE field — spread keeps the rest:
setUser({ ...user, name: 'Ali' });
//         ↑ copies all other fields (email, age)
//                    ↑ overrides only name

// ❌ NEVER do this — loses the other fields:
setUser({ name: 'Ali' }); // email & age are gone!
⚠️

Golden Rule: Never modify state directly. Always use the setter function (setXxx). React only knows about changes when you tell it through the setter.

04 Under the Hood

What Happens on Re-render?

Every time you call a setter function, React goes through a specific process. Understanding this helps you predict how your app behaves.

1
You call setState(newValue) — e.g., setCount(5)
2
React saves the new state value internally for that component
3
React re-runs your component function — all the code inside it runs again
4
React creates a Virtual DOM (a lightweight JS copy of the real DOM)
5
React compares old Virtual DOM vs new Virtual DOM (this is called "diffing")
6
React updates only the parts of the real DOM that changed — nothing else is touched
⚡

Re-renders are not slow! React's diffing algorithm is extremely optimized. It only updates the exact DOM nodes that need changing — not the entire page.

05 Advanced Pattern

Functional Updates

Sometimes the new state depends on the old state. There's a safer way to handle this case.

⚠️ can cause bugs

// Uses the "count" from when
// the component last rendered.
// If React batches updates,
// this can be stale (outdated)!

setCount(count + 1); // ⚠️ risky
setCount(count + 1); // both see same count!
setCount(count + 1); // result: only +1 😱

✅ always correct

// "prev" is ALWAYS the latest
// value, even if multiple
// updates are queued up!

setCount(prev => prev + 1); // ✅
setCount(prev => prev + 1); // ✅
setCount(prev => prev + 1); // ✅ result: +3!

🔁 The Cleanest Toggle Pattern

toggle.jsx
const [isOpen, setIsOpen] = useState(false);

// ✅ Best toggle pattern — always uses latest value:
const toggle = () => setIsOpen(prev => !prev);

return (
  <>
    <button onClick={toggle}>
      {isOpen ? 'Close' : 'Open'}
    </button>
    {isOpen && <div>Modal content here</div>}
  </>
);
🧠

Think of prev => prev + 1 as saying: "Whatever the value is right now, add 1 to it." This is safer than referring to a potentially outdated variable.

06 Controlled Inputs

Forms & Controlled Inputs

In React, form inputs should be "controlled" — meaning the input's value always comes from state, and any change updates state. This gives you a Single Source of Truth.

🔍 SearchBox Example

SearchBox.jsx
function SearchBox() {
  const [query, setQuery] = useState('');

  return (
    <div>
      <input
        type="text"
        value={query}         // ← reads FROM state
        onChange={(e) => setQuery(e.target.value)}
        // ↑ e.target.value = what the user typed
        placeholder="Search..."
      />
      {query && <p>You typed: "{query}"</p>}
      <button onClick={() => setQuery('')}>
        Clear
      </button>
    </div>
  );
}

📝 Multiple State in One Component

UserForm.jsx
function UserForm() {
  // Each field gets its own state — clean and simple
  const [name,        setName]        = useState('');
  const [email,       setEmail]       = useState('');
  const [age,         setAge]         = useState(0);
  const [isSubmitted, setIsSubmitted] = useState(false);

  const handleSubmit = () => {
    console.log({ name, email, age });
    setIsSubmitted(true);
  };

  if (isSubmitted) {
    return <div>✅ Welcome, {name}!</div>;
  }

  return (
    <div>
      <input value={name} onChange={e => setName(e.target.value)} />
      <input value={email} onChange={e => setEmail(e.target.value)} />
      <button onClick={handleSubmit}>Submit</button>
    </div>
  );
}
07 TypeScript

useState with TypeScript

TypeScript adds type safety to your state. Most of the time it figures out the type automatically, but sometimes you need to be explicit.

types.tsx
// TypeScript infers types automatically for simple values:
const [count, setCount] = useState(0);     // type: number ✅
const [name,  setName]  = useState('');    // type: string ✅

// When it could be null or undefined, be explicit:
const [user, setUser] = useState<User | null>(null);

// For arrays, tell TS what's inside:
const [todos, setTodos] = useState<Todo[]>([]);

// Define your types with an interface:
interface Todo {
  id: number;
  text: string;
  completed: boolean;
}
